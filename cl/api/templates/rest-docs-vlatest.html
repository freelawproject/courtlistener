REST API – v3.15

Getting Started & Overview

For developers that wish to have a granular API for our data, we provide a RESTful API based on the Django REST Framework.

This API provides responses in JSON, XML, or HTML, with JSON as the default. If you visit the API using your browser you'll receive nicely formatted HTML making your work as a developer as simple as possible. Hit this button, and you'll open a new tab listing all of the endpoints this API supports:

Show Endpoints

We could have also pulled up that same information using curl, using a command like:

curl "{% get_full_host %}{% url "api-root" version="v3" %}"

Note that when you press the button in your browser, you got an HTML result, but when you run curl you got a JSON object. This is deliberate and is discussed in more depth below.

Watch out! Something else that's curious just happened. You didn't authenticate to the API. This is deliberate to encourage experimentation, but the biggest gotcha people run into is that they forget to enable authentication before deploying their code. Don't let this be you! Remember to add authentication.

The endpoints listed in this response can be used in combination to make sophisticated queries against our database or our search engine. To learn more about an endpoint, you can send an OPTIONS request to it, like so: 

curl -X OPTIONS "{% get_full_host %}{% url "api-root" version="v3" %}"

An OPTIONS request is typically the best way to understand the API, however as of v3.1 we also offer an experimental swagger/OpenAPI interface. These endpoints can be used with a variety of bindings to incorporate the API into your code.

Data Modeling

The two images below show how these endpoints work together. The first image shows the models we use for people and the second shows the models we use for documents and metadata about them. You can see that these models currently link together on the Docket, Person, and Court tables. (Here's a version of this diagram that shows everything all at once.)

In general the basic ideas are:

  • Dockets are the top-level objects and they contain Opinion Clusters, Audio files, and Docket Entries. Dockets are the highest level object in the API, serving to group other objects together.
  • Opinion Clusters are collections of Opinion objects. Opinions can be Dissents, Concurrences or other types of opinion, and so this is how they are grouped together for a specific case. If we don't know if an Opinion is a Dissent, Concurrence or something else, we just call it a Combined Opinion. You can see all the different types of Opinions by sending an OPTIONS request to the Opinion endpoint.
  • Docket Entries are the individual rows of data on a PACER docket. In PACER some rows on the docket refer to several files, and so Docket Entries contain RECAP Document objects to represent those sub-resources. Thus, the big picture is, Dockets contain Docket Entries. Docket Entires contain RECAP Documents.
  • As a general rule, metadata lives at the highest level where it makes sense. For example, it's possible to put docket numbers in the Opinion Cluster objects, but it makes more sense at a higher level—at the Docket level. A sometimes-confusing counter example are case names, which have metadata at the Docket level and at the Opinion Cluster level. This is because case names can change at different times, and because a single Docket could have multiple Opinion Clusters, each with a different case name. See more information about case names.
  • Each Docket is associated with a single jurisdiction. Generally, this is the court where the dispute was litigated. If the case moves between jurisdictions, each jurisdiction has its own Docket.
  • As of version three of this API, Opinions objects cite each other, allowing a dissent to cite a concurrence or any other combination of Opinion types.
  • Judges have Careers, Titles, Positions, and a number of other fields as described in the image above.

Authentication

Authentication is necessary so we can monitor usage of the system and so we can assist with any errors that may occur. Per our privacy policy, we do not track your queries in the API, though we do collect statistical information for system monitoring.

There are currently three types of authentication available on the API:

  1. HTTP Basic Authentication
  2. Django Session Authentication (using cookies)
  3. Django REST Framework Token Authentication

One form of authentication that is currently not supported is OAuth2. If this is a valuable form of authentication for your service, please get in touch using our contact form, or chime in to the issue where we're tracking this feature. If there is enough demand, or if somebody steps forward to develop this feature, it will be added.

All communication with the CourtListener servers is over HTTPS, and so the choice of which form of authentication to use is generally a question of what's most convenient for the context of your work. Our recommendations are:

  • Use Basic Authentication for testing.
  • Use Django Session Authentication if you already have a user's cookie or are developing a system where you can ask the user to log into CourtListener.
  • Use Token Authentication for all normal programmatic API access.

Until now, in this documentation we've used the API without authenticating at all. We made this possible for tutorials like this, but there are very strict throttles for users that are not authenticated. It is not recommended to use the APIs without authenticating. (See more about Throttles.)

HTTP Basic Authentication

To do HTTP Basic Authentication using curl, you might do something like this: 

curl --user "harvey:your-password" "{% get_full_host %}{% url "opinioncluster-list" version="v3" %}"

You can also do it in your browser with a url like:

{% get_full_host username='harvey' password='your-password' %}{% url "opinioncluster-list" version="v3" %}

But if you're using your browser, you might as well just log in.

Django Session Authentication

To use Django Session Authentication, simply log into Courtlistener.

Token Authentication

To use your token, use the Authorization HTTP Header. The key should be prefixed by the string literal Token, with whitespace separating the two strings. For example: 

Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}

Using curl, this looks like:

curl "{% get_full_host %}{% url "opinioncluster-list" version="v3" %}" --header 'Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}'

Note that quotes are used to enclose the whitespace in the header.

Careful! The most common mistake we see is forgetting the word "Token" in the token authentication header value. Remember that the header name is Authorization and the value is Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}

Serialization Formats

Requests may be serialized as HTML, JSON, or XML, with JSON as the default if no format is specified. The format you wish to receive may be requested via the HTTP Accept header.

The following media types and parameters can be used:

  • HTML: The media type for HTML is text/html.
  • JSON: The media type for JSON is application/json. Providing the indent media type parameter allows clients to set the indenting for the response, for example: Accept: application/json; indent=2.
  • XML: The media type for XML is application/xml.

JSONP, a clever browser hack, is no longer provided due to the wide support of CORS. YAML is no longer supported due to the ubiquitous availability of JSON parsers.

Note that by default, browsers send an Accept header similar to: 

text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8

This states that text/html is the preferred serialization format, and the API respects that, returning HTML when requested by a browser and returning JSON when no Accept header is provided (such as when using cURL).

If you wish to set the Accept header using cURL, you can do so using the --header flag: 

curl --header "Accept: application/xml" "{% get_full_host %}{% url "opinioncluster-list" version="v3" %}"

All data is serialized using the utf-8 charset.

Parsing Uploaded Content

If you are a user that has write access to these APIs (most users do not), you will need to use the Content-Type HTTP header to explicitly set the format of the content you are uploading. The header can be set to any of the values that are available for serialization or to application/x-www-form-urlencoded or multipart/form-data, if you are sending form data.

Filtering

Each endpoint can be filtered in a variety of ways using a technique similar to Django's field lookups (see exceptions in footnote 1). The best way to see how an endpoint can be filtered is by doing an OPTIONS request on the endpoint, and inspecting the filters key in the response.

In the filters key, you'll find a dictionary listing all of the fields that can be used for filtering along with their their types, lookup fields, and any values (aka choices) that can be used for specific lookups. For example, every Opinion Cluster has a field called lexis_cite that can be used for lookups. An OPTIONS request describes it like so: 

"lexis_cite": {
    "type": "CharFilter",
    "lookup_types": [
        "exact"
    ]
}

This means that it is a CharFilter, and that it can only be filtered for exact matches.

Here's a more complicated example:

"scdb_votes_majority": {
    "type": "NumberFilter",
    "lookup_types": [
        "exact",
        "gte",
        "gt",
        "lte",
        "lt",
        "range"
    ]
}

This field, scdb_votes_majority, is a NumberFilter which allows you to filter by exact matches, numbers that are greater than (gt), greater than or equal (gte), less than (lt), less than or equal (lte), or within a range.

These field lookups can be used by appending them to a field with a double underscore as a separator. For example, this would return all clusters with a scdb_votes_majority value greater than 2: 

curl "{% get_full_host %}{% url "opinioncluster-list" version="v3" %}?scdb_votes_majority__gt=2"

These filters can be combined using multiple GET parameters.

There are a few special types of filters. The first are RelatedFilters, which allow you to use any of the filters from one object on a different, related object. For example, when you are using the Docket endpoint, you'll see that it has a filter described as: 

"court": {
    "type": "RelatedFilter",
    "lookup_types": "See available filters for 'Courts'"
}

This means that you can use any of the Court filters while filtering Dockets. For example, the Court endpoint advertises this filter: 

"id": {
    "type": "CharFilter",
    "lookup_types": [
        "exact"
    ]
}

Thus, you can lookup dockets from a specific jurisdiction using:

curl "{% get_full_host %}{% url "docket-list" version="v3" %}?court__id=scotus"

This opens up many possibilities. For example, another filter on the Court endpoint is for jurisdictions: 

"jurisdiction": {
    "type": "MultipleChoiceFilter",
    "lookup_types": [
        "exact"
    ],
    "choices": [
        {
            "display_name": "Federal Appellate",
            "value": "F"
        },
        {
            "display_name": "Federal District",
            "value": "FD"
        },
        ...more choices here...
    ]
}

This is a MultipleChoiceFilter, which means you can select multiple values. In the choices key (truncated above), it lists values that you can filter on.

Thus, if you want to filter for Dockets that occur in Federal Appellate (F) or Federal District (FD) courts, you can do: 

curl "{% get_full_host %}{% url "docket-list" version="v3" %}?court__jurisdiction=F&court__jurisdiction=FD"

Naturally, related filter queries and lookup types can be combined to arrive at queries like: 

curl "{% get_full_host %}{% url "docket-list" version="v3" %}?court__date_modified__gt=2013-01-01T00:00:00Z"

Which would return all dockets that occurred in a court that was modified in our system after January 01, 2013. Note that date-times are described in the OPTIONS response as IsoDateTimeFilters, and that dates are simply DateFilters. The former need dates as above, with the complete date and time provided as described by ISO-8601. The latter only need the date to be provided.

RelatedFilters can span many objects and can also be reversed. For example, if you want to get all of the Supreme Court Opinion objects, you will need to do that with a query such as: 

curl "{% get_full_host %}{% url "opinion-list" version="v3" %}?cluster__docket__court__id=scotus"

This uses the Opinion endpoint to get Opinions that are part of Opinion Clusters that are on Dockets in the Court with the ID of scotus. This follows the principle stated above that metadata lives at the highest level possible — it makes more sense for court data to be on the docket, not the cluster or the opinion itself.

A final trick that can be used with the filters is the exclusion parameter. Any filter can be converted into an exclusion filter by prepending it with an exclamation mark. For example, this returns Dockets from non-Federal Appellate jurisdictions: 

curl "{% get_full_host %}{% url "docket-list" version="v3" %}?court__jurisdiction!=F"

You can see more examples of almost any type of filter in our automated tests.

Ordering

To see which fields can be used for ordering, look at the ordering key in an OPTIONS request (see exceptions in footnote 1). For example, the Position endpoint contains this: 

"ordering": [
    "id",
    "date_created",
    "date_modified",
    "date_nominated",
    "date_elected",
    "date_recess_appointment",
    "date_referred_to_judicial_committee",
    "date_judicial_committee_action",
    "date_hearing",
    "date_confirmation",
    "date_start",
    "date_retirement",
    "date_termination"
]

Thus, you can order using any of these fields in conjunction with the order_by parameter.

Descending order can be done using a minus sign. Multiple fields can be requested using a comma-separated list. This, for example, returns judicial Positions ordered by the most recently modified, then by the most recently elected: 

curl "{% get_full_host %}{% url "position-list" version="v3" %}?order_by=-date_modified,-date_elected"

Ordering by fields with duplicate values is non-deterministic. If you wish to order by such a field, you should also provide a tie-breaker in order to consistently order results. For example, ordering by date_filed will not return consistent ordering for items that have the same date, but this can be fixed by ordering by date_filed,id. In that case, if two items have the same date_filed value, the tie will be broken by the id field.

Field Selection

To save bandwidth and increase serialization performance, fields can be limited by using the fields parameter with a comma-separated list of fields you wish to receive.

For example, if you wish to only receive the educations and date_modified fields from the Judge endpoint you could do so with: 

curl "{% get_full_host %}{% url "person-list" version="v3" %}?fields=educations,date_modified"

You can also exclude fields using fields!=educations,date_modified.

Unfortunately, this cannot be used for nested resources, though there is an open issue tracking this.

Limitations & Throttles

At present, this API is throttled to 5,000 queries per hour. Anonymous users (those that are not logged in), are allowed 100 queries per day for testing purposes.

These numbers are counted by using the X-Forwarded-For and Remote-Addr HTTP headers, which can create problems for users that are behind a gateway that uses Network Address Translation (NAT). For more information about how this works, read the documentation on the Django REST Framework website, but in general, the simplest solution is usually to log in. Note that using the browsable HTML API counts towards these thresholds since it's technically just another mediatype of the API.

If you find that you are hitting a throttle while using token authentication, check if it is resolved by browsing the API while logged into your browser. If so, this indicates that the token authentication in your code is not working properly and you are being throttled as an anonymous user. You can also see your recent API requests by looking in your profile, but remember that will show stats for the browsable API as well.

If after checking the above you need your rate limit increased, please get in touch so we can discuss your use case.

Performance Tips

A few things to consider that may increase your performance:

  1. When doing deep crawls of the data, going to very high page numbers will incur increasingly bad performance. This is common in databases because to go to a high page number means sorting the entire data set, then counting to the correct location. Page 50 isn't a big deal. Page 2,000 starts getting slower.

    The fix to this solution is often to work with "slices" of the data. Instead of paginating across the whole result set, use a date field or another range-type field to only paginate over a smaller set of the data. For example, you could slice a large result set by the month of the date filing field to prevent deep pagination.

    Be careful to slice using a field with a normal distribution. Do not use one like date_created, which could have extreme spikes of activity.

  2. Avoid doing queries like court__id=xyz when you can instead do court=xyz. Doing queries with the extra __id introduces a join that can be very expensive.

  3. In general less data is easier and faster to get than more. Could you use a field to filter your result set down to something smaller?

  4. When using the search endpoint, smaller result sets are faster. It isn't always possible to tweak your query to return fewer results, but sometimes it is possible to do a more precise query first, thus making a more broad query unnecessary. For example, a search for an individual in their expected jurisdiction will be faster than doing it in the entire corpus, and may be worthwhile even if it fails and requires the second, broader query some percentage of the time. This will benefit from profiling in your use case and application.

Downloading Files

API responses will give you the URL path to a resource such a PDF or MP3, but not the domain. To download a resource, create a full URL by concatenating the path from the API response with either https://www.courtlistener.com/ or https://storage.courtlistener.com/. Historically, we kept these resources on our own servers, and they were accessible at the www subdomain.

Recently, we transitioned our content to AWS S3, which serves them from the storage subdomain. Prefer using the storage subdomain if you find that the www subdomain is redirecting you. If you are being redirected, that indicates that you are using a URL that is deprecated and that is being maintained for backwards compatibility.

Case Names

Unfortunately, case Names are a complicated topic, and there are a few common misconceptions about them:

  1. Case names are not unique. There's nothing stopping a court from using Johnson v. State every time a person named Johnson sues the State. Indeed, this is common.
  2. Case names are not stable. Many cases begin their lives by suing the governor or the Attorney General. But these people change as a result of elections or appointments and so the names of these cases must change too.
  3. Case names don't always contain v. Instead, many cases start with In Re or other abbreviations.

We've done our best to normalize case names wherever we can, following Blue Book conventions. Some things we regularly do include:

  1. We normalize all the different ways people write "United States" (U.S.A., United States of America, U.S., etc.) All of these variations simply become "United States", as the Blue Book requires.
  2. We remove some abbreviations, like Et al.
  3. We replace any instance of vs. with v.
  4. We do our best to make the case names Title Case rather than UPPERCASE, as many courts insist on publishing them.

The result of all this work is that we have pretty good case names.

In the API, you'll see case names in a variety of places due to the fact that they can change over time. For example, there are case_name fields on the Docket object and also on the Opinion Cluster objects. In almost all instances, these values are the same but some cases have different values for the Docket than for an Opinion Cluster, so we keep this data in duplicate.

Case names can also vary considerably in length, and we've identified three different types of case names:

  1. case_name: This is a typical case name that can be used in almost all references. An example might be Roe v. Wade.
  2. case_name_short: This is a shortened one or two word case name that can be used in legal writing after the standard case name has been provided by the author. An example of this might be simply Roe used, "As was stated in Roe…"
  3. case_name_full: This is the full list of plaintiffs and defendants in a case, sometimes going on for hundreds of words, and often including titles or other information. For example: Roe Et Al v. Wade, District Attorney of Dallas County.

All cases have at least one of these values, many have two values and some have all three. A case_name_short was generated algorithmically for every case possible, but about 40% of cases were too complicated for our program to process and so they were left blank. Human editors have ensured that every Supreme Court case from 1945 to present has a case_name_short value.

case_name is available for the majority of cases, except those that were imported from Resource.org, which only provided case_name_full values. When that happened, there was no way for us to condense a case_name_full, so until a human editor is available to review them, they will lack this value. All cases have either a case_name or a case_name_full value.

Our general-purpose advice is to use a case_name if it is available and to fall back on the case_name_full if it is not. If you are interested in sponsoring trained humans to improve this data, please get in touch.

The absolute_url Field

The absolute_url field shows the URL where a resource can be seen live on the site. It is absolute in the sense that it should never change and will always be backwards compatible if changes are ever needed.

In some cases you might start with a URL from CourtListener, and you might want to look up the item in the API. Generally, CourtListener URLs are designed to contain an ID that can be used in the API to look it up, following a pattern like: 

/$document-type/$numeric-id/$name-of-the-case/

There are three sections:

  1. $document-type: This is the type of document that has been returned, for example, "opinion" indicates that you have gotten an opinion as your result.
  2. $numeric-id: This is a numeric representation of the ID for the document. This value increments as we add content to the system. Note that due to deletions and modifications the numeric IDs are not guaranteed to be purely sequential—IDs will be missing.
  3. $name-of-the-case: This is the "slug" of the document, and generally mirrors its case name. This value can change if we clean up a case name, but provided it is not omitted completely, this part of the URL can be any value without ever affecting the page that is loaded. Put another way, we load pages based on the $numeric-id and $document-type, not by the name of the case.

Docket objects link Opinion Cluster, Audio, and Docket Entry objects together, following the conventional definition of a docket. As with all other endpoints, you can look up the field descriptions, filtering options, ordering options, and rendering options by making an OPTIONS request.

Docket Entry, Party, and Attorney objects are not listed on docket instances due to performance reasons. Some dockets have tens of thousands of docket entries and hundreds of parties. {% url 'originatingcourtinformation-list' version="v3" %}

Originating Court Information represents the information gathered at an appellate court about the case when it was in a lower court or administrative body. The information in this table is joined via a one-to-one relationship to the Docket object. Generally, this table is only completed for appellate cases that we acquire from PACER.

Some systems may normalize this data by linking the lower and upper courts on the database level rather than storing this information in a separate table, but CourtListener does not. Because we get information about the lower court case from the appellate court website, we prefer to keep it separate. This keeps the information acquired from each court compartmentalized so that when we get data from one case, it does not affect the data in the other.

Cross-walking from the upper court docket to the lower is still possible using the the docket_number and appeal_from fields.

{% url 'fjcintegrateddatabase-list' version='v3' %}

  • The various codebooks for civil, criminal, appellate, and bankruptcy datasets.

The field, filtering, and ordering parameters for Docket Entry and RECAP Document objects can be found via an OPTIONS request.

{% url "opinioncluster-list" version="v3" %}

This is a major endpoint serving up the millions of Opinion Clusters that are available on CourtListener. As with all other endpoints, you can look up the field descriptions, filtering options, ordering options, and rendering options by making an OPTIONS request.

Follows are a few notes on the fields that are available:

  • There are presently two fields, judges and panel that contain similar information. The judges field is simply a string field containing the names of judges that heard a case. The panel field is a foreign key to the Judges endpoint, providing more detail about a specific judge or judges. These two fields exist because we have not historically had access to good judicial data, forcing us to simply save the name of the judge as a string.

  • There are numerous citation fields such as federal_cite_one, neutral_cite, state_cite_two, etc. As of API version 3.6, these are all deprecated fields, replaced by the citation field, which lists the same information, but in a more normalized format. These fields will stop getting new values on February 1, 2019, and be removed from the API on May 1, 2019. Both the old and the new approach aim to hold all of the possible parallel citations that an Opinion Cluster may have. To see the types of citation we recognize, do an OPTIONS request on this endpoint.

    Some opinions have multiple citations of a given type. In our old schema, we supported this by having several fields of that type. For example, federal_cite_one, federal_cite_two, and federal_cite_three, which together can hold three parallel federal citations for a single opinion. Our new system handles this much more intuitively.

    Finally, you can learn more about citation types in our Reporter Database project.

  • The source field is a very important one, as it designates where the content of the Opinion Cluster came from. This is vitally important when you are looking at the sub_opinions of the Opinion Cluster (the dissents, concurrences, etc.) because the source field indicates whether we will have the plain text or HTML of the item.

    • html_with_citations is a special field that is populated by parsing one of the above fields for citations, generating an HTML file with hyperlinked citations. All items should eventually have this field, though it can be empty initially or if the cross-linker crashes. In general, this is the field that is used to generate pages on CourtListener and the one we recommend.
    • html_columbia will be populated if we got the content from the Columbia collaboration.

{% url "search-list" version="v3" %}

This endpoint allows you to query our search engine in a similar manner to our front end. Because this endpoint does not use the Django models directly, this endpoint is quite different than any other. Unlike other endpoints, it does not support OPTIONS requests and thus is not self-documenting, however this endpoint is easy to use because GET parameters from the front end can simply be copy-pasted onto this endpoint. For example, this URL on the front end: 

{% get_full_host %}{% url "show_results" %}?q=foo

Becomes this, on the API:

{% get_full_host %}{% url "search-list" version="v3" %}?q=foo

Several Additional differences should also be noted:

  • Ordering: Instead of using a minus sign to flip ordering, it uses asc and desc.

    In addition, results may be sorted randomly by using random_{your-seed} desc/asc as your sort parameter. It's important to notice that {your-seed} must be an integer to get results that are consistent across pagination. If a non-integer value is used, the system will default to using the current timestamp as the seed. This means results will vary for each page load.

    For example:

    {% get_full_host %}{% url "search-list" version="v3" %}?q=foo&order_by=random_123+desc

    This way of sorting the results also works on the front end, but does not show up via the dropdowns.

  • Filtering: Instead of using the Django lookup fields, it uses Solr syntax. To filter on this endpoint, we recommend identifying an effective query on the front end that generates the results you want and then using that query (and variations thereof) to query the API. On the backend, very similar code serves both the API and the front end.

    GET parameters from the front end can be copy-pasted to the API endpoint.

  • Fields: The fields on this endpoint often differ slightly from those in the other endpoints. Both their names and their values may be different. For example, instead of having a field called nature_of_suit, this endpoint has a field called suitNature. Instead of having a number of citation fields (federal_cite_one, state_cite_two, etc.), this endpoint just has all the citations in a single field called citations, which provides the same values as a list.

    These changes are all necessary to make this endpoint a powerful and user friendly search engine. For example, users can query natures of suit using a query like suitNature:foo instead of having to write underscores. Users can search all citations at once, not having to think about whether it's a federal citation or a state one or something else.

    You can see the field definitions on the Advanced Search Help Page.

  • This endpoint can return different kinds of objects, depending on the type parameter. Currently supported parameters are:
    • o — Opinions
    • r — Document-oriented results from the RECAP Archive
    • d — Docket-oriented results from the RECAP Archive
    • p — Judges
    • oa — Oral Arguments
  • As in the front end, when the type is set to return opinions, only precedential results are returned by default. To get other precedential statuses, you need to explicitly request them.
  • The r type returns a flat list of document results from the RECAP Archive, unlike as is done in the front end, where the results are nested. This is to simplify the backend and to aid in performance of API requests. The d type returns a flat list of docket results from the RECAP Archive.

Some fields warrant more details:

  • snippet: This field contains the same values as are found on the front end, utilizing the HTML5 <mark> element to identify up to five matches in a document. If you wish to use the snippet but do not want highlighting, simply use CSS to give the mark element no styling, like a span element. This field only responds to arguments provided to the q parameter. If that parameter is not used, the snippet field will show the first 500 characters of the text field.
  • stat_*: For this Boolean field, we provide opinionated default values. Because most searchers are not interested in non-precedential (unpublished) results, we leave them out by default. If you wish to receive these items, you must explicitly ask for them as you do on the front end.

Financial Disclosure Endpoints

This endpoint provides an API for financial disclosures of current and former federal judges beginning in 1987. You can learn more about which disclosures are included and the limitations of this data source on our coverage page for financial disclosures.

Background on the Information

Judicial officers and certain judicial employees in the United States are required to file financial disclosure reports by Title I of the Ethics in Government Act of 1978. The Act requires that designated federal officials disclose publicly their personal financial interests, to ensure confidence in the integrity of the federal government by demonstrating that they are able to carry out their duties without compromising the public trust.

The Act enumerates the types of information required and prescribes the general format and procedures for the reports. Our APIs mirror the Act's language, with APIs corresponding to each required disclosure type:

  • Financial Disclosures — {% url "financialdisclosure-list" version="v3" %}

    This endpoint contains information about the main document itself and is the link between the other financial disclosure endpoints and the judges themselves.

  • Positions — {% url "disclosureposition-list" version="v3" %}

    This endpoint lists the positions held as an officer, director, trustee, general partner, proprietor, representative, executor, employee, or consultant of any corporation, company, firm, partnership, trust, or other business enterprise, any nonprofit organization, any labor organization, or any educational or other institution other than the United States.

  • Agreements — {% url "agreement-list" version="v3" %}

    Agreement or arrangement of the filer in existence at any time during the reporting period.

  • Non-Investment income — {% url "noninvestmentincome-list" version="v3" %}

    Source, type, and the actual amount or value of earned or other non-investment income aggregating $200 or more from any one source that is received.

  • Non-Investment income (Spouse) — {% url "spouseincome-list" version="v3" %}

    The source and type earned of non-investment income from the spouse of the filer of the financial disclosure.

  • Reimbursements — {% url "reimbursement-list" version="v3" %}

    The identity of the source and a brief description (including travel locations, dates, and nature of expenses provided) of any travel-related reimbursements aggregating more than $415 in value that are received by the filer from one source during the reporting period.

  • Gifts — {% url "gift-list" version="v3" %}

    The source, a brief description, and the value of all gifts aggregating more than $415 in value that are received by the filer during the reporting period from any one source.

  • Debts — {% url "debt-list" version="v3" %}

    All liabilities specified by that section that are owed during the period beginning on January 1 of the preceding calendar year and ending fewer than 31 days before the date on which the report is filed.

  • Investments — {% url "investment-list" version="v3" %}

    The source and type of investment income, including dividends, rents, interest, capital gains, or income from qualified or excepted trusts.

General Notes on Financial Disclosures

  1. We attempted identify all redactions within each disclosure. When a line in the disclosure contains a redaction, it will usually have a redacted field present and set to True. This is your hint that you may want to investigate that row more carefully.

  2. Investments, Debts,and Gifts use form based value codes to indicate monetary ranges instead of exact values. Place an OPTIONS request to these endpoints to learn the values of those fields.

  3. Investment objects contain the field has_inferred_values. This field indicates that we inferred an investment description based on the style of data entry. For example, an investment could have been bought in Q1, while a dividend was paid out in Q2 before being sold in Q4. Often, after the first entry of the investment, subsequent rows in the table are mostly blank. In this instance, we infer the values.

    The table below gives a brief example where we would infer that the blank cell below the cell for AAPL also refers to AAPL:

    Description Date Type
    AAPL 2020-01-01 Bought
    — 2020-02-01 Sold

    In this (slightly contrived) instance our DB would have two rows in the Investment table. The first would be for the purchase of the AAPL stock, and the second would be for the sale of it.

Useful Examples

One can query for investments by stock name, transaction dates and even gross values. For example, the following query is for financial disclosures with individual investments valued above $50,000,000.00. Note that this uses a value code as explained in the general notes above:

curl "{% get_full_host %}{% url "investment-list" version="v3" %}?gross_value_code=P4&fields=investments"

Additionally, one could pinpoint gifts of individual judges when combining the gift database with our judicial database. The following query returns all reported gifts by the late Ruth Bader Ginsburg (her ID is 1213):

curl "{% get_full_host %}{% url "financialdisclosure-list" version="v3" %}?person=1213&fields=gifts"

RECAP Endpoints

The RECAP endpoints provide three high-level features. They allow you to check if a PACER item exists in CourtListener, they allow you to upload items to CourtListener for processing, and they allow you to request that we fetch items from PACER on your behalf.

Some of these endpoints require authentication with a CourtListener account that has special permissions. To obtain permission to use these endpoints, please get in touch.

The endpoints for RECAP are:

  • {% url "processingqueue-list" version="v3" %} — This endpoint is used to upload items and to check on the progress of an upload.

  • {% url "fast-recapdocument-list" version="v3" %} — This endpoint is used to check if documents are in RECAP.

  • {% url "pacerfetchqueue-list" version="v3" %} — Use this endpoint to ask our server to fetch an item from PACER on your behalf.

{% url "processingqueue-list" version="v3" %}

This endpoint can be used to upload PACER content to the RECAP Archive. We describe the process for completing these uploads below, and you can see examples of them in CourtListener's automated test suite. Uploads to these endpoints should be done using HTTP POST requests and multipart form data.

The following parameters apply to all uploads:

  • upload_type (required) — This field accepts integers representing objects in PACER. Send an HTTP OPTIONS request to this API to learn the possible values for this field. As a simple example, 1 represents HTML dockets, 3 represents PDFs, etc.

  • filepath_local (required) — Use this field to upload the binary data you are submitting, whether it HTML of a docket or attachment menu or a PDF file.

  • court (required) — The CourtListener court id, as described here. These are almost all identical to the subdomains of the PACER websites with the exception of the following values that vary due to historical reasons: 

    PACER_TO_CL_IDS = {
    'azb': 'arb',         // Arizona Bankruptcy Court
    'cofc': 'uscfc',      // Court of Federal Claims
    'neb': 'nebraskab',   // Nebraska Bankruptcy
    'nysb-mega': 'nysb'   // Remove the mega thing
};

  • debug (optional) — While you are developing, use this field to test your work. When it is set to true, your uploads will not make changes to the RECAP Archive, but you will create processing requests which will be processed in debug mode.

The following four upload types are supported:

  • District/Bankruptcy Dockets (1), Appellate Dockets (5), Docket History Reports (4), and Claims Registries (9) — Uploading these types of objects should be fairly easy. In addition to the required fields above, supply the pacer_case_id field.

  • PDFs (3) — To upload PDFs, you must include the pacer_doc_id and document_number fields. If you are uploading an attachment, you must also provide the attachment_number field. Because some cases share documents, the pacer_case_id field should also be provided, though it's not a required field if it's unknown.

    pacer_doc_id is the number you see in URLs when purchasing documents on PACER and in the HTML when clicking document numbers on docket pages. For example, in the URL ecf.flp.uscourts.gov/doc1/035021404350, the pacer_doc_id is 035021404350.

    There are three variables embedded in each pacer_doc_id number.The first three digits (in this case, 035) are a code indicating the court. The fourth digit is a zero or one, and is used by PACER to determine if URL should load an attachment page for the document or instead take you directly to the purchase page (we believe this digit is why the URL mentions /doc1/). The remaining digits are the serial number of the document itself. When uploading to this endpoint, the fourth digit must always be normalized to a zero before uploading.

    The document_number is the number for the document that you see on the docket page and elsewhere. The attachment_number is the number you see on the attachment page and elsewhere.

    PDF uploads will only succeed when they can be associated with a docket. If the RECAP Archive does not already have a docket for the pacer_doc_id you uploaded, your upload will be re-queued and retried several times in the hopes that the docket gets processed in the meantime. If, after retrying several times, the docket cannot be found, your upload will be marked as an orphan document. Later, if the docket is found, your PDF will be automatically associated with it, but until then it's not visible to users or any API.

  • Document Zips (10) — From the attachment page in district court PACER websites, there is a link to get all the documents for a particular docket entry as a zip. Such zips can be uploaded using the same parameters as PDFs, using the upload_type of 10.

  • Attachment Menus (2) — These are the HTML pages that you will see that list the attachments for a docket entry. The only required field for this upload type is pacer_case_id.

When you complete your upload, you will create a Processing Queue object in the CourtListener database. This object will be returned in the HTTP response to your upload so you will know its ID. This object will contain the fields you uploaded, and the following fields will be populated as the item is processed:

  • status — When you upload an item, it is only placed into a queue until processing resources are available to merge it into the RECAP Archive. Use this field to determine where in that process your item is. To see the possible values, place an OPTIONS request to this endpoint.
  • error_message — This field will provide you information about whether your upload was processed successfully or will explain any errors that occurred.
  • docket, docket_entry, and recap_document — After an item is successfully processed, these fields will be populated with the IDs of the items that were created or updated. The docket field will be populated for dockets that were created or updated, and all three fields will be populated for uploaded PDFs.
  • date_created and date_modified
An Example

Pulling this all together, a docket upload might look like:

curl -v \
    --form upload_type=1 \
    --form "filepath_local=@docket.html" \
    --form court=dcd \
    --form pacer_case_id=<some-value> \
    --form debug=true \
    '{% get_full_host %}{% url "processingqueue-list" version="v3" %}'
    --header 'Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}'

And once you've completed your upload, you would have a Processing Queue object that you could monitor for completeness. Once its status field indicated that it was processed successfully, it would have a link to the docket that was created or updated.

{% url "fast-recapdocument-list" version="v3" %}

To check if a document is in RECAP, simply complete a query like so: 

curl '{% get_full_host %}{% url "fast-recapdocument-list" version="v3" %}?docket_entry__docket__court=dcd&pacer_doc_id__in=04505578698,04505578717' \
    --header 'Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}'

The ID fields in that query are the CourtListener court abbreviation and the document ID numbers as found in PACER. Several document ID numbers can be provided, separated by commas. A maximum of 300 items will be returned per page.

Take care: The fourth digit of a PACER document ID can be a zero or one. We always normalize it to zero and you will need to do so as well in your queries. For more on these IDs, see the Juriscraper Project.

To query whether a case is in the RECAP Archive, use the Docket endpoint described above.

{% url "pacerfetchqueue-list" version="v3" %}

Use this API to add PACER content to our system so that it is available via our website, APIs, and replicated database. This API can currently use your PACER credentials to download dockets, PDFs, or attachment pages from PACER.

Because downloading content from PACER takes time, this API is asynchronous. After it receives a POST request, it immediately responds with an ID for the request and it enqueues the request for downloading. Typically, requests are fulfilled within seconds.

As the request is processed, it will have one of several statuses:

  • (1) Awaiting processing in queue
  • (2) Item processed successfully
  • (3) Item encountered an error while processing
  • (4) Item is currently being processed
  • (5) Item failed processing, but will be retried
  • (6) Item failed validity tests
  • (7) There was insufficient metadata to complete the task

To monitor your request, you can either poll the API for your request, or you can use our Fetch Webhook endpoint to get immediate updates without polling. We recommend using the webhook endpoint, since it reduces traffic and load on our servers.

Dockets are more complicated than PDFs or attachment pages because there are a handful of options available when purchasing dockets on PACER. Accordingly, we explain the simpler PDFs and attachment pages first.

To purchase a PDF:

  1. Set request_type to 2, which indicates PDFs.

  2. Identify the PDF you wish to add to our system. To identify the PDF use the recap_document field to provide the RECAP Document ID of the item you wish to download.

An example of downloading a PDF by recap_document ID might be: 

curl -X POST \
    --data 'request_type=2' \
    --data 'pacer_username=xxx' \
    --data 'pacer_password=yyy' \
    --data 'recap_document=112' \
    --header 'Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}' \
    "{% get_full_host %}{% url "pacerfetchqueue-list" version="v3" %}"

If you have a client code, you can provide it to the API with the client_code parameter.

A known issue we currently have is that if we do not have the pacer_doc_id for a particular recap_document, we will not be able to download it. If that's the case, you'll get an error message asking you to download the docket, which will get us the pacer_doc_id we need.

Attachment pages are the pages that you see in PACER after you click to download a document, if that document has attachments. These pages are free in PACER. Fetching attachment pages is done same as PDFs, above, but with request_type set to 3.

Fetching dockets is done similarly:

  1. Provide the request_type of 1 for dockets.

  2. Indicate the docket you want by either a CourtListener docket ID, a pacer_case_id-court pair or a docket_number-court pair:

    • pacer_case_id is the internal ID in the PACER system.

    • docket_number is the visible docket number humans use to refer to the case.

    • court is the CourtListener court ID, not the PACER court ID (see the mapping above under PACER_TO_CL_IDS).

  3. Set any options you need for the docket purchase itself (place an HTTP OPTIONS request to this API for details on the available fields).

For example, this request identifies a case by docket number and court:

curl -X POST \
  --data 'request_type=1' \
  --data 'pacer_username=xxx' \
  --data 'pacer_password=yyy' \
  --data 'docket_number=5:16-cv-00432' \
  --data 'court=okwd' \
  --header 'Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}' \
  "{% get_full_host %}{% url "pacerfetchqueue-list" version="v3" %}"

This is the same, but includes parties and counsel:

curl -X POST \
  --data 'request_type=1' \
  --data 'pacer_username=xxx' \
  --data 'pacer_password=yyy' \
  --data 'docket_number=5:16-cv-00432' \
  --data 'court=okwd' \
  --data 'show_parties_and_counsel=true' \
  --header 'Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}' \
  "{% get_full_host %}{% url "pacerfetchqueue-list" version="v3" %}"

For the full list of options on a docket request, do an HTTP OPTIONS request to the endpoint.

Finally, this request updates an existing docket in CourtListener by its ID, but only gets the parties and counsel. Docket entries are excluded by requesting only ones from before 1980: 

curl -X POST \
  --data 'request_type=1' \
  --data 'pacer_username=xxx' \
  --data 'pacer_password=yyy' \
  --data 'docket=5' \
  --data 'show_parties_and_counsel=true' \
  --data 'de_date_end=1980-01-01' \
  --header 'Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}' \
  "{% get_full_host %}{% url "pacerfetchqueue-list" version="v3" %}"

A few general notes:

  1. Whenever a docket is created or updated, we look for PDFs that we already have from that case, and we link them up to the newly created or updated docket. As a result, some docket updates may immediately have PDFs associated with them.

  2. Your first request in a while will take longer than subsequent requests because we will need to gather your PACER cookies by logging you in. Subsequent requests within the same hour will use your cached cookies.

  3. As explained in the security section below, we cache your PACER cookies once we get them. As a result, if we already have your cookies due to a successful previous request, we may not throw an error if your credentials are wrong on a subsequent request — after all, we never save your password, and since we have the cached cookies, we don't need to test the credentials you just sent.

Security of RECAP Fetch API

A security maxim is to never share your password. This API requires that you violate this maxim. Why should you do so, and how do we handle your password securely?

While we prefer not to have unhashed user passwords, PACER lacks any permissions-based or granular authentication system. This means that the only way we can act on your behalf is to have your credentials.

Once we have your password, we work to rid ourselves of it as quickly as possible. We do not store it in our database or logs at any time. Instead, we use it to immediately log into the PACER system. That gives us cookies for your account, which we store in our in-memory database with a one hour expiration period. As soon as we have the cookies, we throw away your username and password.

The result of this system is that we have your password only until we have logged you in, and after that point, we only have a cache of your cookies for one hour.

This API gets content on your behalf using your access rights. This means that if you use this API to request a sealed item from PACER, we will go get it and add it to our system, just like you asked. Do not do this. If you do this accidentally, please get in touch as soon as possible so we can revert the error.

If you have questions about our approach, please see our vulnerability reporting policy and bug bounty program, where you'll find details on contacting us.

These endpoints allow you to programmatically see and create visualizations of Supreme Court networks in CourtListener. All visualizations are associated with a user and are private by default. When you GET these endpoints, you will see data for visualizations that have been made public by their owners or that you have created yourself.

To create a new visualization, POST a title, a starting cluster ID and an ending cluster ID to {% url "scotusmap-list" version="v3" %}:

curl -X POST \
  --data 'cluster_start=/api/rest/v3/clusters/105659/' \
  --data 'cluster_end=/api/rest/v3/clusters/111891/' \
  --data 'title=A map from Trop to Salerno' \
  --header 'Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}' \
  "{% get_full_host %}{% url "scotusmap-list" version="v3" %}"

Note that the cluster_start and cluster_end parameters use URLs instead of IDs. This is in keeping with our API, which uses hyperlinks as identifiers.

Once POSTed, a visualization will be created unless there are no connections between the start and end clusters or the network becomes too large to generate. Once created, the visualization will have nested JSON data representing the visualization itself, a list of clusters that are in it, and various other metadata.

Changing data for an existing item can be done via PATCH requests. For example, to make an item published (so it's publicly viewable), you can do:

curl -X PATCH \
  --data 'published=True' \
  --header 'Authorization: Token {% if user.is_authenticated %}{{ user.auth_token }}{% else %}<your-token-here>{% endif %}' \
  "{% get_full_host %}{% url "scotusmap-detail" version="v3" pk="1" %}"

Similar commands can be used to update other non-readonly fields if desired.

Visualizations can be marked as deleted by flipping the deleted field to True, or can be deleted forthright via a DELETE request.

Maintenance Schedule

Generally, we keep the APIs up and running around the clock, but for the rare times where maintenance requires downtime, we will schedule that downtime during a weekly maintenance window from 21:00-23:59PT on a Thursday night. If you are scheduling cron jobs or otherwise crawling the API, you may experience downtime during this window.

Additionally, we regularly perform bulk tasks on our servers and maintain a public calendar for tracking them. You may encounter larger delays while bulk processing jobs are running.
